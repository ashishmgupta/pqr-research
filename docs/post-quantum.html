Post-Quantum Cryptography

Once I understood what was broken, the next question was what replaces it and why the replacement is actually trustworthy. This section covers lattice problems, the NIST standards that came out of an eight-year competition, and how ML-KEM slots into the TLS handshake you already know without changing anything else.

Why Lattice Problems Resist Quantum Computers

My first question when I started looking at this was: if RSA and ECDH are broken, what replaces them, and why is the replacement any better? The answer is that post-quantum algorithms are built on completely different mathematical problems — ones that Shor's algorithm simply does not apply to. The leading approach is lattice-based cryptography, and the intuition behind it is easier to grasp than I expected.

🔷 What is a Lattice? (The honest non-mathematician explanation)

Picture a regular grid of dots — like graph paper. Now imagine that same grid but in 1,000 dimensions instead of 2. That is a lattice. The hard problem is this: given that grid, find the shortest path between two points. In 2D that is trivial. In 1,000 dimensions it becomes genuinely computationally intractable — and crucially, nobody has found a quantum shortcut for it the way Shor found one for factoring. That is the mathematical foundation ML-KEM is built on.

RSA / ECDH — Classical Algorithms

Security based on integer factorization and discrete logarithm. Both are solved efficiently by Shor's algorithm on a quantum computer.

Broken by quantum

ML-KEM — Post-Quantum

Security based on Module Learning With Errors (MLWE) — a lattice problem. No quantum algorithm analogous to Shor's is known for lattice problems. Best known quantum attack gives minimal advantage.

Quantum-resistant

NIST Post-Quantum Standards (2024)

NIST ran an open competition from 2016 to 2024 — eight years of global cryptographers trying to break each other's submissions. The three algorithms that survived and got standardised cover the two things you need to replace in TLS: key exchange and digital signatures. For most network engineers right now, only the first one is immediately relevant.

Key Encapsulation

ML-KEM

FIPS 203

Replaces RSA and ECDH in TLS key exchange. Three parameter sets: ML-KEM-512, ML-KEM-768 (recommended), ML-KEM-1024. This is what protects your HTTPS sessions today.

Digital Signatures

ML-DSA

FIPS 204

Replaces RSA and ECDSA for code signing, certificates, and authentication. Still being adopted — TLS certificates still use classical signatures today. This is the next wave of migration.

Hash-Based Signatures

SLH-DSA

FIPS 205

Alternative signature scheme based on hash functions rather than lattices. Slower and larger signatures, but a different mathematical foundation — useful for algorithm diversity.

What is actually deployed today vs. what is coming: ML-KEM (key exchange) is already live in Chrome, Firefox, Cloudflare, and F5 BIG-IP 17.5.1+. Your TLS certificates still use classical RSA or ECDSA — and that is fine for now. Certificates are short-lived and not subject to harvest now, decrypt later attacks the way session traffic is. The certificate migration (ML-DSA) is the next wave, and it is still being worked out across the industry.

ML-KEM in TLS 1.3 — Step by Step

This is where it gets practical. ML-KEM slots into the TLS 1.3 handshake in exactly the place where RSA or ECDH used to be — the key exchange step. The mechanics are a little different though, and understanding them helped me appreciate what is actually being protected. In RSA, you encrypt a secret and send it across. In ML-KEM, the secret is never sent — it is derived independently on both sides from the encapsulation process. Harder to intercept, by design.

1
Client Hello — advertises X25519MLKEM768 → Server
Browser includes X25519MLKEM768 in its supported_groups. It also sends a speculative key share — ML-KEM public key and X25519 public key — so the server can respond immediately without a round-trip if it supports ML-KEM.
2
Server encapsulates the shared secret ← Server
Server uses the browser's ML-KEM public key to encapsulate a random shared secret. This produces a ciphertext. Server sends this back in its key_share response.
3
Browser decapsulates — both derive the same session key
Browser uses its ML-KEM private key to decapsulate the ciphertext, recovering the shared secret. Both sides now hold the same secret — without it ever being transmitted. Session keys are then derived via HKDF.
4
AES-256-GCM — unchanged, quantum-safe
All HTTP data encrypted with AES-256 as before. This step is identical to classical TLS.
One thing that tripped me up — HelloRetryRequest: If the client does not send the right key share upfront, the server fires back a HelloRetryRequest asking it to try again with the correct group. That is an extra round-trip before the handshake can complete. Cloudflare's default behaviour ("supported" mode) uses this approach — it works, but adds latency. Switching to "preferred" mode via the Cloudflare API makes it send the ML-KEM key share in the very first ClientHello, skipping that extra round-trip entirely. Worth doing.

Hybrid Mode: X25519MLKEM768 — Belt and Suspenders

When you see X25519MLKEM768 in a TLS negotiation, that is the hybrid group identifier. It is ML-KEM-768 and the classical X25519 running together in the same handshake. I genuinely like this approach — it is the kind of decision I would make as a network engineer.

Why run both algorithms at once?

ML-KEM is new. It has gone through eight years of global scrutiny and come out clean, but it does not have the decades of real-world validation that X25519 has. The hybrid approach means an attacker has to break both simultaneously to compromise the session. If ML-KEM ever turns out to have an unexpected weakness, X25519 still has you covered. If a quantum computer eventually breaks X25519, ML-KEM still has you covered.

It is the network engineering instinct applied to cryptography: redundant paths, no single point of failure. I found it reassuring that the designers thought this way too.

TLS 1.3 — non-negotiable

ML-KEM uses the key_share extension that only exists in TLS 1.3. If any part of your stack has a TLS 1.2 minimum set — a Cloudflare zone, an F5 profile, a load balancer — ML-KEM cannot negotiate. It will not fall back gracefully. It just will not happen. Check your minimum TLS versions first.

Your certificates are fine — for now

The server certificate still uses RSA or ECDSA and that is not a problem today. Certificates prove identity, not confidentiality, and they are short-lived — 90 days typically. They are not worth harvesting for future decryption. Post-quantum certificates using ML-DSA are coming but they are a separate migration and not urgent yet.
