Fundamentals — Pre-Quantum Cryptography

I had to understand what we're replacing before I could understand why we're replacing it

How a TLS Handshake Works Today

Before I could understand what post-quantum cryptography changes, I needed to be really clear on what actually happens during a TLS handshake. I thought I knew — turns out I had a few gaps. The key thing that clicked for me: the handshake is solving two completely separate problems at once. First, key exchange — how do two strangers on the internet agree on a shared secret without anyone intercepting it? And second, data encryption — once they have that secret, how do they use it to protect everything they say to each other? Post-quantum cryptography only touches the first one. I'll come back to why that matters.

🌐 Browser / Client
Chrome, Firefox, Edge…
🖥️ Web Server
Cloudflare · F5 · Origin
1
TCP Handshake — before TLS starts
SYN → SYN-ACK → ACK. A reliable transport connection is established. No encryption yet.
2
Client Hello → Server
Browser sends: supported TLS versions, list of cipher suites it supports (e.g. TLS_AES_256_GCM_SHA384), and its supported key agreement groups (e.g. x25519, P-256). In TLS 1.3 it also sends a speculative key share to avoid a round-trip.
3
Server Hello + Certificate ← Server
Server picks a cipher suite, responds with its public key certificate (RSA or ECDSA signed by a trusted CA). The certificate proves the server's identity. The certificate's public key will be used in the key exchange next.
4
Key Exchange — ⚠️ Quantum-Vulnerable Step
RSA mode: Browser generates a random pre-master secret, encrypts it with the server's public key, and sends the ciphertext. Only the server can decrypt it with its private key.

ECDH mode (~70–80% of modern traffic): Both sides contribute key material via Diffie-Hellman math. The shared secret is computed, never transmitted. Either way — Shor's Algorithm on a quantum computer breaks both methods by recovering the server's private key.
5
Derive Session Keys
Both sides now hold the same shared secret. They independently run it through a key derivation function (HKDF) to produce symmetric encryption keys. No key material crosses the network in this step.
6
AES-256-GCM — ✅ Quantum-Safe
All HTTP traffic (GET requests, HTML, login forms, API calls) is now encrypted with AES-256 using the derived session key. Both sides use the same symmetric key (that's what makes it fast). AES-256 is not vulnerable to quantum computers — Grover's algorithm only halves the effective key length (256 → 128 bits, still secure).
The thing that surprised me: Step 4 and Step 6 are completely separate problems — and quantum computers only break Step 4. Your actual data, sitting in Step 6, is encrypted with AES-256 and that's fine. Post-quantum cryptography replaces Step 4 only. The rest of TLS is untouched. I found this reassuring — it's a targeted fix, not a rebuild from scratch.

RSA: Security by Factoring Difficulty

Here's the bit that actually made RSA click for me. The security isn't magic — it's just a maths problem that's easy in one direction and brutally hard in reverse. Pick two large prime numbers, multiply them together, and you get n. That's fast. Trivially fast. But if I hand you n and ask you to find the two primes I started with? That's the problem that's kept RSA secure for decades. The table below shows just how quickly that difficulty scales up.

Compute n = p × q — this is the public modulus.

The Quantum Threat: Shor's Algorithm

So if factoring large numbers is the problem, what does a quantum computer actually do differently? In 1994, a mathematician named Peter Shor figured out that quantum computers could solve this factoring problem exponentially faster than anything classical — not by trying harder, but by exploiting three properties of quantum mechanics that have no equivalent in traditional computing. I'm not a physicist, but here's how I think about each one.

🔵 Superposition

A qubit exists as 0 and 1 simultaneously until measured. Shor's algorithm evaluates a function at all possible inputs at once, rather than one at a time.

🔗 Entanglement

Qubits become linked — measuring one instantly determines entangled partners. Shor's algorithm uses this to correlate results across the superposition.

〰️ Interference

Quantum algorithms amplify correct answers (constructive interference) and cancel wrong ones (destructive interference). The period — and thus the factors — emerges from this process.

What actually breaks and what doesn't: RSA and ECDH both rely on mathematical problems that Shor's algorithm dismantles on a quantum computer — integer factorization and discrete logarithm respectively. AES-256 is a different story. It doesn't rely on either of those problems, and the best quantum attack against it (Grover's algorithm) only halves the effective key length. 256 bits becomes 128 bits — still fine. So your encrypted data isn't the concern. The key exchange is.

⚠️ Where we actually are in 2025

The largest number ever factored using Shor's Algorithm on real quantum hardware is ~21. To break RSA-2048, you'd need roughly 4,000 logical qubits — which translates to millions of physical qubits once you account for error correction. We're not there yet, and most estimates put that 15–20+ years away.

So why are we talking about this now? Read on.

Harvest Now, Decrypt Later (HNDL)

This is the part that shifted my thinking from "interesting future problem" to "we should be doing something about this now." You don't need a quantum computer to start the attack. You just need storage and patience — and nation-state actors have both in abundance. The idea is straightforward and genuinely unsettling.

1
Capture
Adversary archives encrypted TLS sessions from internet taps, ISP access, or BGP hijacking. This is happening today.
2
Store
Ciphertext is stored. It contains the RSA/ECDH key exchange, which is all they need to later derive the session key.
3
Wait
Adversary waits until a cryptographically capable quantum computer exists (estimated 2035–2040+).
4
Decrypt
Run Shor's Algorithm. Recover the session key. Decrypt the archived HTTPS traffic — login sessions, documents, financial data — retroactively.
Ask yourself: what data are you protecting today that still needs to be secret in 2040? Health records. M&A conversations. Government communications. Trade secrets. If the answer is anything, the migration clock is already running — because the capture is already happening. The timeline to migrate to post-quantum is measured in years, not weeks.

Fundamentals — Post-Quantum Cryptography

What replaces RSA, why it works, and how it slots into the TLS handshake you already know

Why Lattice Problems Resist Quantum Computers

My first question when I started looking at this was: if RSA and ECDH are broken, what replaces them, and why is the replacement any better? The answer is that post-quantum algorithms are built on completely different mathematical problems — ones that Shor's Algorithm simply doesn't apply to. The leading approach is lattice-based cryptography, and the intuition behind it is actually easier to grasp than I expected.

🔷 What is a Lattice? (The honest non-mathematician explanation)

Picture a regular grid of dots — like graph paper. Now imagine that same grid but in 1,000 dimensions instead of 2. That's a lattice. The hard problem is this: given that grid, find the shortest path between two points. In 2D that's trivial. In 1,000 dimensions it becomes genuinely computationally intractable — and crucially, nobody has found a quantum shortcut for it the way Shor found one for factoring. That's the mathematical foundation ML-KEM is built on.

RSA / ECDH (Classical)

Security based on integer factorization and discrete logarithm.

Quantum speedup: Shor's algorithm solves these in polynomial time.
Result: Broken by quantum

ML-KEM (Post-Quantum)

Security based on Module Learning With Errors (MLWE) — a lattice problem.

Quantum speedup: none known. Best quantum attack gives minimal advantage.
Result: Quantum-resistant

NIST Post-Quantum Standards (2024)

NIST ran an open competition from 2016 to 2024 — eight years of global cryptographers trying to break each other's submissions. The three algorithms that survived and got standardised cover the two things you actually need to replace in TLS: key exchange and digital signatures. For most network engineers right now, only the first one is immediately relevant.

Key Encapsulation

ML-KEM

FIPS 203

Replaces RSA and ECDH in TLS key exchange. Three parameter sets: ML-KEM-512, ML-KEM-768 (recommended), ML-KEM-1024. This is what protects your HTTPS sessions.

Digital Signatures

ML-DSA

FIPS 204

Replaces RSA and ECDSA for code signing, certificates, and authentication. Lattice-based. Still being adopted — TLS certificates still use classical signatures today.

Hash-Based Signatures

SLH-DSA

FIPS 205

Alternative signature scheme based on hash functions (not lattices). Slower and larger signatures, but a different mathematical foundation — useful for algorithm diversity.

What's actually deployed today vs. what's coming: ML-KEM (key exchange) is already live in Chrome, Firefox, Cloudflare, and F5 BIG-IP 17.5.1+. Your TLS certificates, on the other hand, still use classical RSA or ECDSA — and that's okay for now. Certificates are short-lived and aren't subject to harvest-now-decrypt-later attacks the way session traffic is. The certificate migration (ML-DSA) is the next wave, and it's still being worked out across the industry.

ML-KEM in TLS 1.3 — Step by Step

This is where it gets practical. ML-KEM slots into the TLS 1.3 handshake in exactly the place where RSA or ECDH used to be — Step 4 from the diagram above. The mechanics are a little different though, and understanding why helped me appreciate what's actually being protected. In RSA, you encrypt a secret and send it across. In ML-KEM, the secret is never sent — it's derived independently on both sides from the encapsulation process. Harder to intercept, by design.

🌐 Browser
TLS 1.3 client
☁️ Server
Cloudflare · F5
1
Client Hello — advertises X25519MLKEM768 → Server
Browser includes X25519MLKEM768 in its supported_groups extension. It also sends a speculative key share (ML-KEM public key + X25519 public key) in the key_share extension, so the server can respond immediately if it supports it.
2
Server encapsulates shared secret ← Server
Server uses the browser's ML-KEM public key to encapsulate a random shared secret. This produces a ciphertext (the encapsulated key). Server sends this ciphertext in its key_share response.
3
Browser decapsulates — both derive session key
Browser uses its ML-KEM private key to decapsulate the ciphertext, recovering the shared secret. Both sides now hold the same secret — without it ever being transmitted. Session keys are derived via HKDF.
4
AES-256-GCM — unchanged, quantum-safe
All HTTP data encrypted with AES-256 as before. This step is identical to classical TLS.
One thing that tripped me up — HelloRetryRequest: If the client doesn't send the right key share upfront, the server fires back a HelloRetryRequest asking it to try again. That's an extra round-trip before the handshake can complete. Cloudflare's default behaviour ("supported" mode) uses this approach — it works, but adds latency. Switching to "preferred" mode via the API makes Cloudflare send the ML-KEM key share in the very first ClientHello, skipping that extra round-trip entirely. Worth doing.

Hybrid Mode: X25519MLKEM768 — Belt and Suspenders

When you see X25519MLKEM768 in a TLS negotiation, that's the hybrid group identifier. It's ML-KEM-768 and the classical X25519 running together in the same handshake. I genuinely like this approach — it's the kind of decision I'd make as a network engineer.

Why run both at once?

ML-KEM is new. It's gone through eight years of global scrutiny and come out clean, but it doesn't have the decades of real-world validation that X25519 has. The hybrid approach means an attacker has to break both simultaneously to compromise the session. If ML-KEM ever turns out to have an unexpected weakness, X25519 still has you covered. If a quantum computer eventually breaks X25519, ML-KEM still has you covered.


It's the network engineering instinct applied to cryptography: redundant paths, no single point of failure. I found it reassuring that the designers thought this way too.

TLS 1.3 — non-negotiable

ML-KEM uses the key_share extension that only exists in TLS 1.3. If any part of your stack has a TLS 1.2 minimum set — whether it's a Cloudflare zone, an F5 profile, or a load balancer in front — ML-KEM simply cannot negotiate. It won't fall back gracefully. It just won't happen. Check your minimum TLS versions first.

Your certificates are fine — for now

The server certificate still uses RSA or ECDSA and that's not a problem right now. Certificates prove identity, not confidentiality, and they're short-lived — 90 days typically. They're not worth harvesting for future decryption. Post-quantum certificates (using ML-DSA) are coming but they're a separate migration and not urgent today.

Getting a Working OpenSSL (Required for CLI Verification)

I'll save you an hour of frustration here. When I first tried to verify ML-KEM negotiation from the command line, I hit an error that looked like a server problem but was entirely a client problem. The openssl s_client command needs OpenSSL 3.5.0+ to even know what X25519MLKEM768 is. Older versions just refuse with a hard error — and depending on your shell and OS, you might be running an older version than you think.

The error I hit — and what it actually means:
Call to SSL_CONF_cmd(-groups, X25519MLKEM768:x25519) failed
error:0A080106:SSL routines:gid_cb:passed invalid argument ... group 'X25519MLKEM768' cannot be set

The server is fine. Your openssl binary is too old or it's the wrong one in your PATH. Here's how to sort it out.

Step 1 — Figure out which openssl you're actually calling

Check what you're running

which openssl
openssl version

You need 3.5.0 or later. Anything older doesn't know what X25519MLKEM768 is.

Check available PQ groups — but use the right flag

openssl list -tls-groups | grep MLKEM

I wasted time on this: openssl list -groups returns nothing at all in OpenSSL 3.5. It's not broken — it's just a different command. Use -tls-groups.

Step 2 — Fix by platform

🪟 Windows — Git Bash (MinGW64)

Good news here: Git for Windows already ships with OpenSSL 3.5.x in MinGW64. The catch is you have to run your commands in Git Bash, not PowerShell or CMD. In PowerShell, the openssl command resolves to something older from C:\Windows\System32. Same command, completely different binary. 

# Confirm you are in Git Bash with the right binary
which openssl
# Should show: /mingw64/bin/openssl

openssl list -tls-groups | grep MLKEM
# Should show: X25519MLKEM768

🐧 Ubuntu / WSL (Ubuntu 22.04 or 24.04)

This is where I spent most of my time. Ubuntu ships OpenSSL 3.0.x — which is too old. You need to build 3.5.0 from source. It sounds more involved than it is — about 5 minutes once you have the dependencies. The important thing is the -Wl,-rpath flag below. Without it, the binary builds fine but then fails at runtime because it finds the system's old libraries instead of its own. I hit that exact error and it's confusing until you know why. 

# 1. Install build dependencies
sudo apt install -y build-essential wget

# 2. Download and extract OpenSSL 3.5.0 source
wget https://github.com/openssl/openssl/releases/download/openssl-3.5.0/openssl-3.5.0.tar.gz
tar xzf openssl-3.5.0.tar.gz
cd openssl-3.5.0

# 3. Configure with rpath so it finds its own libraries (important!)
./Configure --prefix=/usr/local/openssl35 \
            --openssldir=/usr/local/openssl35 \
            -Wl,-rpath,/usr/local/openssl35/lib64

# 4. Build and install (~5 minutes)
make -j$(nproc)
sudo make install

# 5. Verify
/usr/local/openssl35/bin/openssl version
# OpenSSL 3.5.0 ...

Without -Wl,-rpath, the binary links against the system's old libssl.so.3 and fails with "version OPENSSL_3.4.0 not found" errors. The rpath flag bakes the library path into the binary so it always finds its own libraries.

🔧 Ubuntu / WSL — already built without rpath? Don't rebuild, just do this

If you already ran make install without the rpath flag, you'll see a "version not found" error when running the binary. No need to start over — LD_LIBRARY_PATH fixes it immediately:

# Prefix every command with the library path
LD_LIBRARY_PATH=/usr/local/openssl35/lib64 \
  /usr/local/openssl35/bin/openssl version

# Or add a shell function to ~/.bashrc for convenience
cat >> ~/.bashrc << 'EOF'
openssl35() {
  LD_LIBRARY_PATH=/usr/local/openssl35/lib64 \
    /usr/local/openssl35/bin/openssl "$@"
}
EOF
source ~/.bashrc

# Then just use:
openssl35 version
openssl35 s_client -connect example.com:443 -tls1_3 \
  -groups "X25519MLKEM768:x25519" -brief

Step 3 — Run the actual test

Once you have a working OpenSSL 3.5+ binary sorted, the command itself is simple and the same everywhere:

# Replace 'openssl' with your binary path / alias if needed
openssl s_client \
  -connect your-hostname:443 \
  -tls1_3 \
  -groups "X25519MLKEM768:x25519" \
  -CAfile /etc/ssl/certs/ca-certificates.crt \
  -brief
